# Kotlin Standards

Kotlin conventions for Entur applications. Read [CONVENTIONS.md](../CONVENTIONS.md) first for cross-language standards, and [java.md](java.md) for shared JVM patterns (Spring Boot, logging, Redis, testing, dependencies, Dockerfile).

## Runtime and Build

- **Kotlin version**: latest stable (currently 2.x)
- **Java target**: 25 (or latest LTS/stable)
- **Build tool**: Gradle with Kotlin DSL (`build.gradle.kts`)
- **Framework**: Spring Boot 3.x with Kotlin support
- **Linting**: Ktlint
- **Dependency management**: Gradle version catalogs (`gradle/libs.versions.toml`)
- **API approach**: Contract-first with OpenAPI Generator (preferred)
- **SQL library**: JetBrains Exposed SQL-DSL (preferred for Kotlin projects)

### build.gradle.kts

```kotlin
plugins {
    alias(libs.plugins.kotlin.jvm)
    alias(libs.plugins.kotlin.spring)
    alias(libs.plugins.spring.boot)
    alias(libs.plugins.spring.dependency.mgmt)
    alias(libs.plugins.openapi.generator)        // For contract-first APIs
}

group = "org.entur"

java {
    toolchain {
        languageVersion = JavaLanguageVersion.of(25)
    }
}

kotlin {
    compilerOptions {
        freeCompilerArgs.add("-Xjsr305=strict")   // strict null-safety with Spring annotations
    }
}

tasks.withType<Test> {
    useJUnitPlatform()
}
```

### Layered Boot JAR

Configure for optimized Docker image caching:

```kotlin
tasks {
    bootJar {
        layered {
            enabled = true
            application {
                intoLayer("spring-boot-loader") {
                    include("org/springframework/boot/loader/**")
                }
                intoLayer("application")
            }
            dependencies {
                intoLayer("internal-dependencies") {
                    include("org.entur*:*:*")
                }
                intoLayer("dependencies")
            }
            layerOrder = listOf("dependencies", "internal-dependencies", "spring-boot-loader", "application")
        }
    }
}
```

## Kotlin-Specific Patterns

### Data Classes for Domain Models

Use `data class` for domain models (not DTOs -- those are generated by OpenAPI). Include default values where appropriate and use nullable types for optional fields.

### Contract-First REST Controllers

With OpenAPI Generator, controllers implement generated interfaces (e.g., `VersionApi`). Add `@PreAuthorize` for authorization. Use a mapper to convert between generated DTOs and domain models.

Without OpenAPI Generator, use standard Spring annotations (`@RestController`, `@RequestMapping`, `@GetMapping`, `@PostMapping`).

### Service Layer

Define service interfaces and implementations separately. Services orchestrate DAOs, perform validation, and throw typed exceptions (e.g., `ElementNotFoundException`).

### Key Kotlin Principles

- Use `data class` for domain models, value objects, and request/response types
- Use primary constructor injection (not `@Autowired`)
- Use Kotlin null-safety instead of `Optional` -- return `T?` not `Optional<T>`
- Use expression-body functions for simple transformations
- Use trailing commas (improves diffs)
- Prefer `val` over `var`
- Use `sealed class`/`sealed interface` for restricted type hierarchies
- Use `object` for stateless singletons (validators, constants)
- Use `internal` visibility for implementation details
- Prefer `when` over `if-else` chains
- Use scope functions (`let`, `apply`, `run`, `also`) idiomatically

### Mapper Pattern (DTO to Domain)

Use `@Component` mapper classes at the API boundary to convert between generated DTOs and domain models. Include `toDomain`, `toDto`, and `toDtoList` functions.

For simpler projects without OpenAPI Generator, use extension functions (e.g., `fun Route.toResponse() = RouteResponse(...)`).

### Database Access with Exposed SQL-DSL

Exposed provides a typesafe SQL DSL without ORM magic.

#### Entity Definition

Define tables as `object` extending `LongIdTable`. Use `varchar`, `timestamp`, `date`, and `.nullable()` for columns. Follow `snake_case` naming for database columns.

#### DAO Pattern

Use `@Repository` classes implementing a `BaseDAO` interface. Wrap operations in `transaction {}`. Use `insertAndGetId`, `selectAll().where {}`, and map `ResultRow` to domain models.

#### Joins and Queries

Define reusable join functions as extension functions on `Join` within entity companion/object scope.

### Input Validation

Use `object` validators for complex business rules. Declare as `internal object` with public validation functions that throw typed exceptions on failure.

### Exception Handling

Define typed exception classes extending `RuntimeException`. Handle with a `@RestControllerAdvice` class using `@ExceptionHandler` methods that map exceptions to appropriate HTTP status codes and structured error response bodies.

### Coroutines (WebFlux)

Only use if the project already uses WebFlux. ALWAYS use either WebFlux or MVC consistently throughout a project. Use `suspend` functions in controllers for coroutine-based handlers.

## Testing in Kotlin

### Test Libraries

- **JUnit 5**: test framework
- **Kotest**: assertions (`shouldBe`, `shouldNotBe`, `shouldHaveSize`, `shouldThrow`)
- **SpringMockK**: mocking for Spring + Kotlin (`@MockkBean`)
- **TestContainers**: dockerized PostgreSQL for integration tests
- **Spring Boot Test**: `@SpringBootTest`, `@WebMvcTest`
- **Entur Auth Test**: `TenantJsonWebToken` for simulating authenticated requests

### Unit Tests

Use backtick method names for readability. Use Kotest assertions (`shouldBe`, `shouldThrow`) and JUnit 5 (`assertDoesNotThrow`, `assertThrows`). Use test data builders (see below) for constructing test objects.

### Controller Tests (WebMvcTest)

Use `@WebMvcTest` with `@MockkBean` for service mocking. Use `TenantJsonWebToken` with `@InternalTenant` / `@TravellerTenant` annotations for simulating authenticated requests (see [authorization.md](authorization.md#test-configuration)).

### Integration Tests (TestContainers)

Create a shared base class pattern:

- `@TestConfiguration` with `@Bean @ServiceConnection` for `PostgreSQLContainer`
- `@SpringBootTest` with `@Import(TestContainersConfig::class)`
- `@ExtendWith(TenantJsonWebToken::class)` for auth
- `@Sql` for test data cleanup between tests

### Test Data Builders

Use builder classes with `withX()` methods returning `apply { ... }` and a `build()` function. Provide sensible defaults for all fields.

### Test Data Files

Use SQL scripts in `src/test/resources/test-data/` for integration test setup. Reference with `@Sql` annotation on test methods.
